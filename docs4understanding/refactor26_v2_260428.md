The objectives of refactoring
-----------------------------

This branch is a refactor.

We want to consider three types of remote reduction strategy and four types
of backend:

Reduction Strategies:

1. Simple: Series of individual chunk requests, handled via a thread pool
2. Aggregated: Concatenated chunk requests, possibly paged with pages handled by a thread pool.
3. Remote: Send the selection to a remote python.
4. Local: fully local within existing stack

Storage Backends:

| Protocol | Format | Available Strategies | Protocol |
| --- | --- | --- | --- |
| ssh | (anything supported by p5rem) |  1,2,3  | p5rem |
| s3/https | (anything supported by fsspec and quacking like pyfive) | 1 or 2, server dependent | http |
| local posix | anything that quacks like pyfive| 1, 2, 3 (1 and 2 for testing) | posix | 
| remote posix | (limited) | (simple) | (experimental) |

For the moment we will test on HDF5, NetCDF4, and pp (via ppfive). Zarr and Kerchunk support
can come in the future.

(Here remote posix means, posix where we do have reduction support in the storage, as opposed to
using posix to load the chunks into memory on the calling system. We do not discuss remote posix any 
further here as the workd done on this is very prototypical, the rest of the work described here
aspires to be production ready.)

In this context quacking like pyfive means:

- we can obtain a chunk index from the file,
- chunks are identical shape (so some versions of zarr will not be supported),
- we can understand and implement - in the server - any compression technique used.

Reductions available: The default set is

- min
- max
- count
- mean
- sum

Desirable additionals include:

- argmax
- argmin
- range

If the remote supports computation, it should be possible to register an arbitrary
python function, and have it execute and return the response. This is effectively
entering into Dask territory, so while it _should be_ possible, we will not
implement that in the near future.


The key difference is that in the first case, the user is responsible for selecting the file type - ppfive, pyfive, p5rem (itself mimicing ppfive or pyfive) - and the active machinery is applied dataset by dataset, we simply exploit the parent files `dataset.id.index`.  In the second case, pyactivestorage has to choose the formatter. That seems unnecessarily difficult. Let's stick with the former.

But then we have to understand the internals of the dataset if it is not an hdf5 dataset (or pp, or p5rem, since we can make them quack like pyfive). THis might require a shimmy over other formats, but we can postpone that.

We could even, with a bit more work inside pyfive and friends, expose the selection to chunks mapping, so that active storage didn't have to replicate that machinery, that would make the first option even more attractive as the active machinery is little more than a harness over other structures to set up the remote chunk requests based on chunk selection completely under the control of the upsteam classes.
