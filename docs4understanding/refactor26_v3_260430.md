The objectives of refactoring
-----------------------------

This branch is a refactor.

We want to consider four types of reduction strategy and four types
of backend:

Reduction Strategies:

1. Simple: Series of individual chunk requests, handled via a thread pool.
2. Aggregated: Concatenated chunk requests, possibly paged with pages handled by a thread pool.
3. Remote: Send the selection to a remote python runtime.
4. Local: Fully local within existing stack.

Storage Backends:

| Access Path | Format | Available Strategies | Transport / Protocol |
| --- | --- | --- | --- |
| ssh | Anything supported by p5rem | 1, 2, 3 | p5rem |
| s3/https | Anything supported by fsspec and quacking like pyfive | 1 or 2, server dependent | http |
| local posix | Anything that quacks like pyfive | 1, 2, 3 (1 and 2 for testing) | posix |
| remote posix | Limited (prototype) | Simple (prototype) | experimental |

For the moment we will test on HDF5, NetCDF4, and pp (via ppfive).
Zarr and Kerchunk support can come in the future.

(Here remote posix means posix where we do have reduction support in the storage,
as opposed to using posix to load the chunks into memory on the calling system.
We do not discuss remote posix any further here as the work done on this is
very prototypical; the rest of the work described here aspires to be
production ready.)

In this context, quacking like pyfive means:

- we can obtain a chunk index from the file,
- chunks are identical shape (so some versions of zarr will not be supported),
- we can understand and implement in the server any compression technique used.

Reductions available: The default set is

- min
- max
- count
- mean
- sum

Desirable additions include:

- argmax
- argmin
- range

If the remote supports computation, it should be possible to register an arbitrary
python function, and have it execute and return the response. This is effectively
entering into Dask territory, so while it should be possible, we will not
implement that in the near future.

Alternative process model
-------------------------

```plantuml
skinparam defaulttextalignment center
start
: (same starting steps);
: open file\n"f = File(fs)";

: apply active to file\na = Active(f);

: select backend and strategy;

: decorate dataset instances with active method;

: select variable\n"v=a['var']"\n(obtains chunk index);
: select and apply method\n"result=v[slice stuff].min()";
end
```

In both cases the user is responsible for selecting
the file type (ppfive, pyfive, p5rem, with p5rem itself mimicking ppfive or pyfive).

In the first case  the active machinery is applied dataset by dataset, exploiting the parent file `dataset.id.index`.

In the second case, we apply all logic at file level and decorate datasets.

There is also a syntax change we can allow as we can always get the actual variable
slice via the parent file. We can handle the "method 1/2/3 etc" via an argument
on the reduction functions, e.g. `result = v[:].min(strategy=1) + u.min(strategy=1)`
(strategy is really only of interest to us, not users.)

In both cases with a bit more work inside pyfive and friends, expose the
selection-to-chunks mapping so active storage does not have to replicate that
machinery.

That machinery is pretty simple (for full chunks) though:

Decision and migration plan
---------------------------

Decision for this refactor phase:

- Keep the current externally visible API behavior in place.
- Continue with dataset-level execution flow (`av.min()[slice stuff]`) for compatibility.
- Build internal abstractions so we can support a future file-level decorated API
  (`v[slice stuff].min()`) without rewriting backend logic.

Migration steps:

1. Stabilize backend and strategy interfaces first (no public API changes).
2. Introduce an internal lazy selection object that can carry selection + operation.
3. Add an opt-in experimental API path for `v[slice].min()` in parallel.
4. Run both API paths against the same backend execution tests.
5. Decide deprecation timeline only after feature and performance parity.

Backend capability contract
---------------------------

Minimum capability contract for a backend/adapter that claims pyfive-like support:

- Open variable by name from a file-like parent.
- Provide variable shape, dtype, and chunk metadata/index.
- Provide chunk access by byte offset and size, or equivalent chunk locator.
- Provide compression/filter metadata needed for decode and reduction.
- Provide missing-data semantics (`_FillValue`, `missing_value`, valid range where present).

Execution strategy contract:

- Input: selection, axis, operation, backend session/context, chunk metadata.
- Output: reduced array/scalar plus optional count payload for composed stats.
- Must preserve deterministic output shape conventions across strategies.

Transport/session contract:

- Explicit session lifecycle (`open`, `close`, and retry/reconnect policy where relevant).
- Clear exception mapping from transport errors to user-facing Active errors.
- No hidden dependence on global state for credentials or endpoint selection.

Failure semantics matrix
------------------------

Expected failure classes should be aligned by backend type:

| Failure case | ssh/p5rem | s3/https with Reductionist | local posix |
| --- | --- | --- | --- |
| Authentication failure | Session/bootstrap error | HTTP auth error | OS/file permission error |
| File not found | Remote file open error | File-not-found from store loader | FileNotFoundError |
| Bad host / endpoint | bootstrap/connect failure | connection/schema error | n/a |
| Service unavailable | session/remote-server unavailable | Reductionist HTTP 5xx | n/a |
| Decode/compression mismatch | protocol/decode error | reduction/decode error | local decode error |

Test policy implications
------------------------

- Local and loopback integration tests should remain default.
- Externally hosted service tests should be marked distinctly so they can be
  excluded when services are unavailable, while still running by default in
  fully connected environments.

Further Musing on APIs
----------------------

Could we support API syntax: `min(active['var'][slice])` or `np.min(active['var'][slice])`
and have them trigger service-side reduction rather than materialising data?

**Python's built-in `min()` cannot be intercepted.** It calls `__iter__` and
compares elements in Python — there is no `__min__` dunder protocol. Any object
returned by `active['var'][slice]` would be fully iterated and its elements
compared locally, defeating the purpose.

**NumPy's `np.min()` is a different story.** NumPy respects the
`__array_function__` protocol (PEP 18-style dispatch). An object that implements
`__array_function__` can intercept `np.min`, `np.max`, `np.mean`, etc. and
redirect them to a backend reduction call without fetching data.

The required design would be:

1. `active['var']` returns a lazy `VariableSelection` (holds variable name and
   dataset reference, no data fetched).
2. `selection[slice]` returns a lazy `SlicedSelection` (holds variable, slice,
   and dataset reference).
3. `SlicedSelection` implements `__array_function__`, intercepting `np.min`,
   `np.max`, `np.mean`, `np.sum` etc. and dispatching them as backend reduction
   calls.
4. `SlicedSelection` also implements `__array__` as a materialise-on-demand
   fallback for operations that cannot be intercepted.

The realistic target syntax is therefore `np.min(active['var'][slice])`, not the
built-in `min(...)`. The built-in is a dead end for lazy server-side computation.
